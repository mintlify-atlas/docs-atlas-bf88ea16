System design questions test your ability to think through end-to-end architecture under constraints. Interviewers are not looking for a single correct answer — they want to see your thought process, the trade-offs you identify, and your familiarity with mobile-specific constraints.

<Note>
For a comprehensive guide to Android system design interviews, read [Android System Design Interviews](https://outcomeschool.com/blog/android-system-design-interviews) by Amit Shekhar.
</Note>

## How to approach a system design question

<Steps>
  <Step title="Clarify requirements">
    Ask questions before designing anything. Distinguish functional requirements (what it must do) from non-functional requirements (scale, latency, offline support, battery usage). Example: "Should the image library support GIFs? What is the expected cache size?"
  </Step>
  <Step title="Define the scope">
    State what you will and will not cover. This shows you can prioritize. "I'll focus on the image loading pipeline and caching strategy. I'll skip the server-side CDN configuration."
  </Step>
  <Step title="Identify the major components">
    Draw or describe the high-level blocks: UI layer, data layer, network layer, storage, background workers. Name what each component is responsible for.
  </Step>
  <Step title="Drill into the critical path">
    Walk through the most important user journey end-to-end. For an image loader: cache lookup → network request → decode → memory cache write → display.
  </Step>
  <Step title="Discuss trade-offs">
    Every design choice has a cost. Disk cache improves offline support but adds latency on cold reads. A large memory cache reduces flicker but increases OOM risk. Say these things out loud.
  </Step>
  <Step title="Address failure cases">
    What happens when the network is unavailable? When the cache is full? When the process is killed mid-download? Interviewers reward candidates who think about edge cases.
  </Step>
  <Step title="Suggest improvements">
    End with what you would improve given more time: prefetching, progressive image decoding, predictive caching based on scroll velocity.
  </Step>
</Steps>

<Accordion title="Design an Image Loading Library">

**Requirements:** Load images from URLs, display them in `ImageView`, cache images to avoid re-downloading, handle memory efficiently.

**Key components:**

1. **Request manager** — accepts a URL and a target `ImageView`. Manages lifecycle (cancels requests when the view is detached).
2. **Memory cache** — an LRU cache of decoded `Bitmap` objects keyed by URL. Fast — O(1) lookup.
3. **Disk cache** — persists compressed image bytes to the filesystem. Survives process death.
4. **Network fetcher** — downloads image bytes using an `HttpURLConnection` or OkHttp.
5. **Decoder** — converts bytes into a `Bitmap`, sampling down to the target dimensions to avoid loading a 4K image into a 100×100 view.
6. **Bitmap pool** — a pool of reusable `Bitmap` objects. When a new bitmap is needed, the decoder checks the pool for a same-size bitmap to overwrite (`BitmapFactory.Options.inBitmap`), avoiding allocation.
7. **Transformation pipeline** — applies transforms (crop, round corners, grayscale) before placing in cache.

**Request flow:**

```
loadImage(url, imageView)
  ↓
Check memory cache → HIT: display immediately
  ↓ MISS
Check disk cache   → HIT: decode + add to memory cache + display
  ↓ MISS
Network fetch      → decode + add to disk + memory cache + display
```

**Memory management:**
- Use `SoftReference` or `WeakReference` for the secondary (evicted) cache tier.
- Listen to `onTrimMemory()` to reduce cache size under memory pressure.
- Always sample bitmaps to `inSampleSize` matching the target view size.

```kotlin
fun calculateInSampleSize(options: BitmapFactory.Options, reqWidth: Int, reqHeight: Int): Int {
    val (height, width) = options.run { outHeight to outWidth }
    var inSampleSize = 1
    if (height > reqHeight || width > reqWidth) {
        val halfHeight = height / 2
        val halfWidth = width / 2
        while (halfHeight / inSampleSize >= reqHeight && halfWidth / inSampleSize >= reqWidth) {
            inSampleSize *= 2
        }
    }
    return inSampleSize
}
```

**Lifecycle handling:** Tie requests to the lifecycle of the host (`Activity`/`Fragment`) using a headless `Fragment` or `DefaultLifecycleObserver` to cancel in-flight requests when the screen is destroyed.

<Accordion title="Design a File Downloader Library">

**Requirements:** Download files from URLs, support pause/resume, show progress, handle failures with retry, support concurrent downloads.

**Key components:**

1. **DownloadManager** — public API; accepts `DownloadRequest` objects (url, destination path, priority).
2. **Queue** — a priority queue of pending requests. Higher-priority downloads start first.
3. **Worker pool** — a `ThreadPoolExecutor` (or `CoroutineDispatcher`) with a configurable concurrency limit (e.g., 3 simultaneous downloads).
4. **Downloader** — performs the actual HTTP GET with `Range` header support for resume.
5. **Storage writer** — writes bytes to a `.tmp` file, renamed to the final name on completion.
6. **Progress notifier** — emits progress via `Flow<DownloadProgress>` or a callback.
7. **Persistence** — stores download state (url, bytes downloaded, status) in a local database so downloads survive app restart.

**Pause and resume with HTTP Range requests:**

```kotlin
// Resume a partial download
val downloadedBytes = File(tmpPath).length()
val connection = URL(url).openConnection() as HttpURLConnection
connection.setRequestProperty("Range", "bytes=$downloadedBytes-")

// Server responds with 206 Partial Content
val inputStream = connection.inputStream
val outputStream = FileOutputStream(tmpPath, /* append = */ true)
```

**Retry strategy:** Exponential backoff — retry after 1s, 2s, 4s, 8s up to a max. Use `WorkManager` for downloads that must complete even if the app is killed.

**Progress tracking:**

```kotlin
data class DownloadProgress(
    val id: String,
    val bytesDownloaded: Long,
    val totalBytes: Long,
    val status: DownloadStatus
) {
    val percent: Int get() = if (totalBytes > 0) ((bytesDownloaded * 100) / totalBytes).toInt() else 0
}
```

<Accordion title="Design a Networking Library">

**Requirements:** Make HTTP GET/POST requests, support interceptors, handle authentication, cache responses, parse responses to typed objects.

**Key components:**

1. **Request builder** — fluent API to construct `Request` (url, method, headers, body).
2. **Interceptor chain** — a list of `Interceptor` objects that wrap the request/response pipeline. Supports logging, authentication token injection, retry.
3. **Connection pool** — reuses TCP connections and HTTP/2 streams (provided by OkHttp in practice).
4. **Cache** — HTTP-level response caching (respects `Cache-Control` headers).
5. **Dispatcher** — manages in-flight requests; enforces max concurrent requests per host.
6. **Converter** — serializes request bodies and deserializes response bodies (JSON → data class).
7. **Call factory** — creates `Call` objects that can be executed synchronously or enqueued asynchronously.

**Interceptor pattern:**

```kotlin
interface Interceptor {
    fun intercept(chain: Chain): Response

    interface Chain {
        fun request(): Request
        fun proceed(request: Request): Response
    }
}

// Auth interceptor example
class AuthInterceptor(private val tokenProvider: TokenProvider) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val original = chain.request()
        val authenticated = original.newBuilder()
            .header("Authorization", "Bearer ${tokenProvider.getToken()}")
            .build()
        return chain.proceed(authenticated)
    }
}
```

**Coroutine support:** Wrap the async `Call` in a `suspendCancellableCoroutine`:

```kotlin
suspend fun <T> Call<T>.await(): T = suspendCancellableCoroutine { cont ->
    enqueue(object : Callback<T> {
        override fun onResponse(call: Call<T>, response: Response<T>) {
            if (response.isSuccessful) cont.resume(response.body()!!)
            else cont.resumeWithException(HttpException(response))
        }
        override fun onFailure(call: Call<T>, t: Throwable) {
            cont.resumeWithException(t)
        }
    })
    cont.invokeOnCancellation { cancel() }
}
```

<Accordion title="Design a Caching Library (LRU Cache)">

**Requirements:** Cache key-value pairs with a maximum capacity; evict the least recently used item when capacity is reached.

**LRU Cache implementation using `LinkedHashMap`:**

```kotlin
class LruCache<K, V>(private val capacity: Int) {

    private val map = object : LinkedHashMap<K, V>(capacity, 0.75f, /* accessOrder = */ true) {
        override fun removeEldestEntry(eldest: MutableMap.MutableEntry<K, V>?): Boolean {
            return size > capacity
        }
    }

    @Synchronized
    fun get(key: K): V? = map[key]

    @Synchronized
    fun put(key: K, value: V) {
        map[key] = value
    }

    @Synchronized
    fun remove(key: K): V? = map.remove(key)

    @Synchronized
    fun size(): Int = map.size
}
```

**`LinkedHashMap` with `accessOrder = true`** keeps entries in the order they were last accessed. The eldest entry (the one at the head, least recently accessed) is removed when the map exceeds capacity.

**Two-tier cache (memory + disk):**

```kotlin
class TieredCache<K, V>(
    private val memoryCache: LruCache<K, V>,
    private val diskCache: DiskCache<K, V>
) {
    fun get(key: K): V? {
        return memoryCache.get(key) ?: diskCache.get(key)?.also {
            memoryCache.put(key, it)  // promote to memory
        }
    }

    fun put(key: K, value: V) {
        memoryCache.put(key, value)
        diskCache.put(key, value)
    }
}
```

Android's `android.util.LruCache` in `androidx.collection` uses this same approach.

<Accordion title="Design an Analytics Library">

**Requirements:** Track events with properties, batch events before sending to reduce network calls, survive process death, respect privacy settings.

**Key components:**

1. **Event tracker** — public API: `Analytics.track("button_click", mapOf("id" to "checkout"))`.
2. **Event model** — `AnalyticsEvent(name, properties, timestamp, sessionId, userId)`.
3. **Event buffer** — holds events in memory until flushed.
4. **Persistence layer** — writes buffered events to a local database immediately to survive crashes.
5. **Batch uploader** — sends a batch of N events (or after T seconds, whichever comes first) to the server.
6. **Retry manager** — retries failed uploads with exponential backoff using `WorkManager`.
7. **Opt-out manager** — checks user consent before logging any event.

**Flush strategy:**

```kotlin
class AnalyticsDispatcher(
    private val buffer: EventBuffer,
    private val uploader: EventUploader
) {
    private val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())

    init {
        // Flush every 30 seconds or when buffer reaches 50 events
        scope.launch {
            while (isActive) {
                delay(30_000)
                flush()
            }
        }
    }

    fun track(event: AnalyticsEvent) {
        buffer.add(event)
        if (buffer.size >= 50) flush()
    }

    private fun flush() {
        val events = buffer.drain()
        if (events.isNotEmpty()) {
            scope.launch { uploader.upload(events) }
        }
    }
}
```

**Privacy:** Provide `Analytics.optOut()` that clears stored events and sets a flag that prevents future tracking. Check the flag on every `track()` call.

<Accordion title="Design a Logging Library">

**Requirements:** Log messages at different levels (DEBUG, INFO, WARN, ERROR), write to multiple outputs (Logcat, file, remote server), be zero-overhead in release builds.

**Key components:**

1. **Logger API** — `Logger.d("tag", "message")`, `Logger.e("tag", "message", throwable)`.
2. **Log level filter** — discard messages below the configured minimum level.
3. **Tree (appender) pattern** — each `Tree` implementation writes to a specific sink (Logcat, file, Crashlytics).
4. **File writer** — appends to a rotating log file; rotates when file exceeds a size limit.
5. **Remote reporter** — batches `ERROR`/`CRASH` logs and sends to a backend for monitoring.

```kotlin
// Timber-inspired design
abstract class Tree {
    abstract fun log(level: Level, tag: String, message: String, throwable: Throwable?)
}

object Logger {
    private val trees = mutableListOf<Tree>()
    var minimumLevel: Level = Level.DEBUG

    fun plant(tree: Tree) { trees.add(tree) }

    fun d(tag: String, message: String) = log(Level.DEBUG, tag, message, null)
    fun e(tag: String, message: String, t: Throwable? = null) = log(Level.ERROR, tag, message, t)

    private fun log(level: Level, tag: String, message: String, t: Throwable?) {
        if (level >= minimumLevel) trees.forEach { it.log(level, tag, message, t) }
    }
}

// Concrete tree
class LogcatTree : Tree() {
    override fun log(level: Level, tag: String, message: String, throwable: Throwable?) {
        when (level) {
            Level.DEBUG -> Log.d(tag, message, throwable)
            Level.INFO  -> Log.i(tag, message, throwable)
            Level.WARN  -> Log.w(tag, message, throwable)
            Level.ERROR -> Log.e(tag, message, throwable)
        }
    }
}
```

In release builds, plant only the file/remote tree. In debug builds, add the `LogcatTree`.

<Accordion title="Design WhatsApp (messaging with real-time delivery)">

**Requirements:** Send and receive text messages, show delivery receipts (sent / delivered / read), support offline message queuing, enable group messaging.

**Architecture overview:**

- **Transport layer** — persistent WebSocket connection to a message server. Re-establishes automatically on reconnect.
- **Message queue** — messages written locally (pending state) before being sent. On send failure, they remain in the queue and are retried when connectivity is restored.
- **Message store** — local SQLite/Room database storing all messages, conversations, and contacts. The database is the single source of truth for the UI.
- **Sync manager** — on app start / reconnect, fetches any messages received while offline from the server.
- **Receipt system** — the server sends delivery receipts (`delivered`, `read`) back to the sender when the recipient's device receives or opens the message.

**Message delivery flow:**

```
Sender types message
  → Write to local DB (status: PENDING)
  → Send via WebSocket to server
  → Server stores and forwards to recipient
  → Server ACKs sender (status: SENT)
  → Recipient device receives → ACK to server (status: DELIVERED)
  → Recipient opens conversation → ACK to server (status: READ)
  → Server notifies sender of each status change
```

**Offline handling:** If the WebSocket is disconnected, buffer outgoing messages in a local queue (WorkManager). Re-send when connectivity is restored. Fetch missed incoming messages via a REST endpoint on reconnect.

**End-to-end encryption:** Generate a keypair per device. Exchange public keys via the server. Encrypt message content with the recipient's public key before transmission. The server never has access to plaintext.

</Accordion>

<Accordion title="Design Uber / a ride-sharing app">

**Requirements:** Match riders with drivers in real-time, show live driver location on map, handle ride lifecycle (request → match → pickup → en-route → complete).

**Key components:**

- **Location service** — a foreground service for drivers that sends GPS coordinates to the server every 3–5 seconds via WebSocket or HTTP POST.

**Driver location smooth animation:**

```kotlin
fun animateMarkerTo(marker: Marker, newLatLng: LatLng) {
    val start = marker.position
    ValueAnimator.ofFloat(0f, 1f).apply {
        duration = 1000L
        interpolator = LinearInterpolator()
        addUpdateListener { animation ->
            val fraction = animation.animatedFraction
            val lat = start.latitude + fraction * (newLatLng.latitude - start.latitude)
            val lng = start.longitude + fraction * (newLatLng.longitude - start.longitude)
            marker.position = LatLng(lat, lng)
        }
        start()
    }
}
```

**Offline considerations:** Cache last-known driver position. Display a "Reconnecting…" banner when WebSocket drops. Resume with full state sync on reconnect.

<Accordion title="Offline-first app architecture">

An offline-first app works fully without a network connection and syncs changes when connectivity is restored.

**Core principle:** The local database is the single source of truth. The UI always reads from the database; the network layer only writes to the database.

**Architecture layers:**

```
UI ← observes → Local Database (Room)
                     ↑
              Repository
                     ↑ (when online)
              Network API (Retrofit)
```

**Sync strategies:**

- **Pull-based:** Fetch from server on app open and on periodic `WorkManager` tasks.
- **Push-based:** Server sends push notifications to trigger a sync.
- **Conflict resolution:** Use timestamps or vector clocks to decide which version of a record wins when both client and server have changes.

```kotlin
class ArticleRepository(
    private val api: ArticleApi,
    private val dao: ArticleDao
) {
    // UI always observes the local database
    fun observeArticles(): Flow<List<Article>> = dao.observeAll()

    // Sync runs independently
    suspend fun sync() {
        try {
            val remoteArticles = api.getArticles()
            dao.upsertAll(remoteArticles)
        } catch (e: IOException) {
            // Sync failed — UI still shows cached data
        }
    }

    // Local write queued for upload
    suspend fun createArticle(article: Article) {
        val localId = dao.insert(article.copy(syncStatus = SyncStatus.PENDING))
        try {
            val remote = api.createArticle(article)
            dao.update(remote.copy(id = localId, syncStatus = SyncStatus.SYNCED))
        } catch (e: IOException) {
            // Will retry via WorkManager
        }
    }
}
```

</Accordion>

<Accordion title="HTTP Request vs Long-Polling vs WebSocket vs Server-Sent Events">

These are four different strategies for client-server communication, each with different trade-offs.

| Strategy | Direction | Connection | Use case |
|---|---|---|---|
| HTTP Request | Client → Server | New per request | Standard REST API calls |
| Long-Polling | Bidirectional (emulated) | Held open, re-opened | Simple real-time when WebSocket isn't available |
| WebSocket | Bidirectional | Persistent, full-duplex | Chat, live collaboration, gaming |
| Server-Sent Events | Server → Client | Persistent, one-way | Live feed, stock ticker, notifications |

**HTTP Request:** The client sends a request; the server responds immediately. Simple, stateless, scales well. No real-time capability.

**Long-Polling:** The client sends a request; the server holds it open until it has new data to return (or until timeout). The client immediately sends another request after receiving a response. Simulates real-time but wastes connections under low-traffic conditions.

**WebSocket:** A single TCP connection upgraded from HTTP. Both sides can send messages at any time. Lowest latency for bidirectional real-time communication. Requires persistent connection management and reconnection logic.

**Server-Sent Events (SSE):** HTTP-based. The server streams events to the client over a single long-lived connection. The client cannot send data back on the same connection. Native browser support; simpler than WebSocket for one-way push.

**In Android:**

```kotlin
// WebSocket with OkHttp
val client = OkHttpClient()
val request = Request.Builder().url("wss://api.example.com/ws").build()
val ws = client.newWebSocket(request, object : WebSocketListener() {
    override fun onMessage(webSocket: WebSocket, text: String) {
        // handle incoming message
    }
    override fun onFailure(webSocket: WebSocket, t: Throwable, response: Response?) {
        // reconnect with exponential backoff
    }
})
```

<Accordion title="Voice and video call architecture">

Voice and video calling requires ultra-low latency and is fundamentally different from data APIs.

**Key technologies:**

- **WebRTC** — open standard for real-time peer-to-peer audio/video. Used by Google Meet, Discord, and most mobile calling apps.
- **Signaling server** — a WebSocket or HTTP server used to exchange WebRTC session description protocol (SDP) offers/answers and ICE candidates between peers. It does not carry media.
- **STUN server** — helps a device discover its public IP address and NAT type.
- **TURN server** — relays media when a direct peer-to-peer connection cannot be established (e.g., both peers are behind symmetric NATs).

**Call setup flow:**

```
Caller                    Signaling Server          Callee
  |                              |                     |
  |-- Create SDP offer --------->|                     |
  |                              |-- Forward offer ---->|
  |                              |<-- SDP answer -------|
  |<-- Forward answer -----------|                     |
  |-- Exchange ICE candidates ---|-- Forward -------->  |
  |<----- Direct P2P Media --------------------------------->|
```

**Audio quality:** Use the Opus codec (default in WebRTC). Enable acoustic echo cancellation (AEC) and noise suppression — both built into WebRTC's audio processing pipeline.

**Android integration:** Use the `libwebrtc` Android AAR or a managed SDK (Twilio, Agora, Vonage) to avoid maintaining the native WebRTC stack.

</Accordion>

<Accordion title="Real-time update options in Android">

| Option | Latency | Battery | Complexity | Best for |
|---|---|---|---|---|
| HTTP Polling | High | Poor | Low | Infrequent updates |
| Long-Polling | Medium | Medium | Medium | Fallback when WS unavailable |
| WebSocket | Low | Medium | High | Chat, live data, gaming |
| Server-Sent Events | Low | Good | Medium | One-way server push |
| Firebase Realtime DB / Firestore | Low | Good | Low | Rapid prototyping |
| FCM (Push) | Medium | Excellent | Medium | Background notifications |

**Choosing the right option:**

- Use **WebSocket** when you need bidirectional, low-latency communication (chat, multiplayer).
- Use **SSE** when the server pushes updates to the client but the client doesn't need to respond on the same channel (news feed, live scores).
- Use **FCM** to wake the app and trigger a sync; do not use it to deliver large payloads.
- Use **polling** only as a last resort or for infrequent background updates.

</Accordion>

<Accordion title="Network optimization options for mobile apps">

Mobile networks are unreliable, high-latency, and metered. Key optimization strategies:

**Reduce request count:**
- Batch multiple small requests into one.
- Use GraphQL to fetch exactly the fields you need.
- Cache aggressively with `Cache-Control` headers.

**Reduce payload size:**
- Use Protobuf or FlatBuffers instead of JSON (30–80% smaller).
- Enable gzip compression (`Accept-Encoding: gzip`).
- Use appropriate image formats: WebP instead of PNG, progressive JPEG for photos.

**Reduce connection overhead:**
- Reuse connections (HTTP/2 multiplexing, OkHttp connection pool).
- Enable HTTP/2 to multiplex multiple requests over one TCP connection.

**Handle connectivity changes:**
- Use `ConnectivityManager.NetworkCallback` to react to network changes.
- Queue requests when offline; flush on reconnect.

**Reduce background data usage:**
- Use `JobScheduler` / `WorkManager` with `NetworkType.UNMETERED` to defer large syncs to Wi-Fi.
- Respect `Data Saver` mode via `ConnectivityManager.getRestrictBackgroundStatus()`.

```kotlin
val networkCallback = object : ConnectivityManager.NetworkCallback() {
    override fun onAvailable(network: Network) {
        // Network restored — trigger pending sync
        syncManager.triggerSync()
    }
    override fun onLost(network: Network) {
        // Network lost — queue outgoing requests
        requestQueue.pause()
    }
}
connectivityManager.registerDefaultNetworkCallback(networkCallback)
```

</Accordion>

<Accordion title="Data syncing on unstable networks">

Unstable networks cause partial writes, duplicate submissions, and inconsistent states. Strategies to handle this:

**Idempotency keys:** Generate a unique ID for each mutation before sending. The server ignores duplicate submissions with the same key.

```kotlin
data class CreateOrderRequest(
    val idempotencyKey: String = UUID.randomUUID().toString(),
    val items: List<OrderItem>
)
```

**Optimistic updates:** Apply the change to the local database immediately, show it in the UI, then sync in the background. If the server rejects it, roll back and show an error.

**Exponential backoff with jitter:**

```kotlin
suspend fun <T> retryWithBackoff(
    maxRetries: Int = 5,
    initialDelayMs: Long = 1000,
    block: suspend () -> T
): T {
    var currentDelay = initialDelayMs
    repeat(maxRetries - 1) { attempt ->
        try {
            return block()
        } catch (e: IOException) {
            val jitter = (0..500).random()
            delay(currentDelay + jitter)
            currentDelay = (currentDelay * 2).coerceAtMost(30_000)
        }
    }
    return block() // last attempt — let it throw
}
```

**Conflict resolution policies:**
- **Last-write-wins** — use server timestamp; most recent update takes precedence.
- **Client-wins** — local changes override server.
- **Manual merge** — present the user with both versions and let them choose.

</Accordion>

<Accordion title="Database Normalization vs Denormalization">

**Normalization** organizes data to reduce redundancy by splitting it into multiple related tables. Each piece of information is stored exactly once.

```sql
-- Normalized: user and orders in separate tables
CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT);
CREATE TABLE orders (id INTEGER PRIMARY KEY, user_id INTEGER, amount REAL,
    FOREIGN KEY (user_id) REFERENCES users(id));
```

- **Pros:** No data duplication; updates in one place propagate everywhere; smaller storage.
- **Cons:** Requires JOINs to reconstruct data, which can be slow for large datasets.

**Denormalization** intentionally duplicates data to speed up reads by avoiding JOINs.

```sql
-- Denormalized: user name stored directly in orders
CREATE TABLE orders (id INTEGER PRIMARY KEY, user_id INTEGER,
    user_name TEXT, amount REAL);
```

- **Pros:** Faster reads; simpler queries.
- **Cons:** Data duplication means update anomalies — if a user renames themselves, all their order rows need updating.

**When to denormalize in Android (Room):**

Use denormalization for read-heavy data that changes infrequently, such as a user's display name in a message history. Normalize write-heavy or frequently updated data to avoid inconsistency.

</Accordion>

<Accordion title="Hash vs Encrypt vs Encode">

These three operations serve different purposes and are not interchangeable.

**Encoding** — transforms data into a different format for compatibility, not security. It is reversible without a key.
- Examples: Base64, URL encoding, UTF-8.
- Use when: transferring binary data over text-only channels.

**Hashing** — one-way transformation of data into a fixed-size digest. It cannot be reversed.
- Examples: SHA-256, bcrypt (for passwords), MD5 (insecure for security use).
- Use when: storing passwords, verifying file integrity, generating cache keys.

**Encryption** — reversible transformation using a key. Only parties with the correct key can decrypt.
- Examples: AES (symmetric), RSA (asymmetric).
- Use when: storing sensitive data at rest, transmitting data securely.

```kotlin
// Hashing a password (use bcrypt in a real app; this is SHA-256 for illustration)
fun hash(input: String): String {
    val bytes = MessageDigest.getInstance("SHA-256").digest(input.toByteArray())
    return bytes.joinToString("") { "%02x".format(it) }
}

// Encoding to Base64
val encoded = Base64.encodeToString("hello".toByteArray(), Base64.DEFAULT)
val decoded = String(Base64.decode(encoded, Base64.DEFAULT))

// Symmetric encryption with AES
val key = KeyGenerator.getInstance("AES").apply { init(256) }.generateKey()
val cipher = Cipher.getInstance("AES/GCM/NoPadding")
cipher.init(Cipher.ENCRYPT_MODE, key)
val cipherText = cipher.doFinal("secret".toByteArray())
```

</Accordion>

<Accordion title="Webhook vs Polling">

**Polling:** The client repeatedly asks the server "is there new data?" at regular intervals.

- Simple to implement.
- Wastes resources when there is nothing new.
- Acceptable for infrequent events (e.g., check for software updates every 24 hours).

**Webhook:** The server calls the client (or a server-side endpoint you register) when an event occurs. The client never has to ask.

- Efficient — no wasted requests.
- Requires the client to expose an endpoint (works server-to-server; not directly applicable to mobile clients without a relay).
- In mobile: FCM acts as the webhook relay — your server sends an event to FCM, FCM delivers a push notification to the device.

**In Android:** Use FCM (Firebase Cloud Messaging) to receive server events without polling. The device does not poll; FCM maintains the persistent connection and wakes the app when a message arrives.

</Accordion>

<Accordion title="WebSocket vs Socket.IO">

**WebSocket** is a network protocol providing full-duplex communication over a single TCP connection. It is a W3C/IETF standard.

**Socket.IO** is a JavaScript library (with client libraries for other platforms including Android) built on top of WebSocket, with additional features:

| Feature | WebSocket | Socket.IO |
|---|---|---|
| Protocol | Standard (RFC 6455) | Custom protocol on top of WS |
| Fallback | None — requires WS support | Falls back to HTTP long-polling |
| Rooms / namespaces | Not built-in | Built-in |
| Auto-reconnect | Not built-in | Built-in |
| Broadcasting | Not built-in | Built-in |
| Binary support | Yes | Yes |
| Android client | OkHttp `WebSocket` | `socket.io-client-java` |

**Choose WebSocket** when you control the server and want a standard protocol with minimal overhead. **Choose Socket.IO** when you need its higher-level features (rooms, namespaces, auto-reconnect) and can accept the dependency.

</Accordion>

<Accordion title="Symmetric vs Asymmetric Encryption">

**Symmetric encryption** uses the same key to encrypt and decrypt. It is fast and suitable for large data.

- Examples: AES-256 (GCM mode preferred), ChaCha20.
- Problem: How do you securely share the key with the other party?

**Asymmetric encryption** uses a key pair: a **public key** (shared openly) to encrypt, and a **private key** (kept secret) to decrypt. Solves the key distribution problem.

- Examples: RSA-2048, EC-Diffie-Hellman.
- Problem: Much slower than symmetric encryption; not practical for large data.

**In practice (hybrid encryption):** Use asymmetric encryption to exchange a symmetric session key, then use the symmetric key for all subsequent data. TLS (HTTPS) works this way.

**Android Keystore:** Store cryptographic keys securely in the hardware-backed Keystore so they cannot be extracted from the device.

```kotlin
// Generate AES key in Android Keystore
val keyGenerator = KeyGenerator.getInstance(KeyProperties.KEY_ALGORITHM_AES, "AndroidKeyStore")
keyGenerator.init(
    KeyGenParameterSpec.Builder("my_key_alias",
        KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT)
        .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
        .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
        .setUserAuthenticationRequired(true)
        .build()
)
val key = keyGenerator.generateKey()
```

</Accordion>

<Accordion title="Query Optimization in SQLite (Room)">

Poorly written queries are a common source of jank. Key optimization techniques:

**Use indexes for frequent WHERE and ORDER BY columns:**

```kotlin
@Entity(indices = [Index(value = ["user_id"]), Index(value = ["created_at"])])
data class MessageEntity(
    @PrimaryKey val id: String,
    val userId: String,
    val content: String,
    val createdAt: Long
)
```

**Select only needed columns:**

```kotlin
// Bad — fetches all columns
@Query("SELECT * FROM messages WHERE user_id = :userId")
fun getMessages(userId: String): List<MessageEntity>

// Good — fetches only what the UI needs
@Query("SELECT id, content, created_at FROM messages WHERE user_id = :userId")
fun getMessageSummaries(userId: String): List<MessageSummary>
```

**Use `LIMIT` and `OFFSET` for pagination:**

```kotlin
@Query("SELECT * FROM messages ORDER BY created_at DESC LIMIT :limit OFFSET :offset")
suspend fun getMessagesPaged(limit: Int, offset: Int): List<MessageEntity>
```

**Analyze with `EXPLAIN QUERY PLAN`:**

```sql
EXPLAIN QUERY PLAN
SELECT * FROM messages WHERE user_id = '123' ORDER BY created_at DESC;
-- If output shows "SCAN" instead of "SEARCH USING INDEX", add an index.
```

**Batch inserts in a transaction:**

```kotlin
@Transaction
suspend fun insertMessages(messages: List<MessageEntity>) {
    messages.forEach { insert(it) }
}
// A single transaction is dramatically faster than N individual inserts.
```

</Accordion>

<Accordion title="Write-Ahead Logging (WAL)">

WAL is a SQLite journaling mode that improves write performance and allows concurrent reads during writes.

**Default journal mode (`DELETE`):** Before modifying a page, SQLite copies the original page to a rollback journal file. Writers block all readers.

**WAL mode:** Instead of overwriting pages in place, SQLite appends changes to a WAL file. Readers can continue reading the last committed state from the main database file while a writer is appending to the WAL.

**Benefits:**
- Reads do not block writes; writes do not block reads.
- Faster write performance in most workloads (no need to sync the journal before writing).
- Allows multiple concurrent readers.

**Enable in Room:**

```kotlin
Room.databaseBuilder(context, AppDatabase::class.java, "app.db")
    .setJournalMode(RoomDatabase.JournalMode.WRITE_AHEAD_LOGGING)
    .build()
```

**Trade-offs:**
- WAL is slightly slower for workloads with very large transactions.
- The WAL file can grow unbounded if checkpointing (merging WAL back into the main database) is delayed.
- Not suitable for databases on network filesystems.

</Accordion>

<Accordion title="Firebase Remote Config">

Firebase Remote Config lets you change the behavior and appearance of your app without publishing a new version.

**How it works:** You define default parameter values in the app. These can be overridden by values fetched from the Firebase console. The app fetches new values in the background and activates them on the next app launch (or explicitly).

```kotlin
// Initialize with defaults
val remoteConfig = Firebase.remoteConfig
remoteConfig.setDefaultsAsync(
    mapOf(
        "enable_new_checkout" to false,
        "max_retry_count" to 3L,
        "banner_text" to "Welcome!"
    )
)

// Fetch and activate
remoteConfig.fetchAndActivate().addOnCompleteListener { task ->
    if (task.isSuccessful) {
        val enableCheckout = remoteConfig.getBoolean("enable_new_checkout")
        val retryCount = remoteConfig.getLong("max_retry_count").toInt()
        // Use values
    }
}
```

**Use cases:** Feature flags, A/B testing, UI copy changes, controlling rollout percentage.

**Caching:** Remote Config caches fetched values locally. The default minimum fetch interval is 12 hours; in debug mode you can set it to 0. Do not set too short an interval in production — Firebase enforces rate limits.

<Tip>
Combine Remote Config with Firebase Analytics audiences to roll out features to specific user segments (e.g., users on Android 12+, users in a specific country).

<Accordion title="SMS Retriever API">

The SMS Retriever API allows your app to automatically read a one-time password (OTP) sent via SMS without requiring the `READ_SMS` permission.

**How it works:**

1. Your app generates a unique hash string tied to the app's signing certificate.
2. Your server sends an SMS with a specific format: starts with `<#>`, ends with the hash.
3. The SMS Retriever API intercepts matching messages and delivers them to your app via a `BroadcastReceiver`.

```kotlin
// 1. Start SMS Retriever
SmsRetriever.getClient(context).startSmsRetriever()

// 2. Register receiver to listen for the SMS
val receiver = object : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        if (SmsRetriever.SMS_RETRIEVED_ACTION == intent.action) {
            val extras = intent.extras
            val status = extras?.get(SmsRetriever.EXTRA_STATUS) as? Status
            if (status?.statusCode == CommonStatusCodes.SUCCESS) {
                val message = extras.get(SmsRetriever.EXTRA_SMS_MESSAGE) as? String
                val otp = message?.let { extractOtp(it) }
                // Auto-fill OTP field
            }
        }
    }
}
registerReceiver(receiver, IntentFilter(SmsRetriever.SMS_RETRIEVED_ACTION))

// SMS format your server must send:
// <#> Your verification code is 123456
// FA+9qCX9VSu  ← 11-char app hash
```

**Benefit:** No sensitive `READ_SMS` permission needed. Google Play and privacy-conscious users prefer this approach.

<Accordion title="Accurate time in Android">

`System.currentTimeMillis()` returns the device's wall clock time, which the user can manually adjust. This is unsuitable for time-sensitive operations (session expiry, event ordering across devices).

**Options for accurate time:**

**1. NTP (Network Time Protocol):** Query a public NTP server to get the true current time, then calculate an offset from the device clock.

```kotlin
// Using the TrueTime library or manual NTP query
class AccurateTime {
    private var offset: Long = 0  // offset from device clock to NTP time

    suspend fun sync() {
        val ntpTime = queryNtpServer("pool.ntp.org")
        offset = ntpTime - System.currentTimeMillis()
    }

    fun now(): Long = System.currentTimeMillis() + offset
}
```

**2. `SystemClock.elapsedRealtime()`:** Returns milliseconds since device boot, including sleep. This clock cannot be adjusted by the user and is suitable for measuring elapsed time between events on the same device.

**3. Server timestamp:** For client-server consistency, always use the server's timestamp as the authoritative time for events. Never trust the client clock for security-sensitive timestamps.

**4. GPS time:** Extremely accurate but requires location permission and GPS hardware — rarely appropriate.

**Best practice:** Use `SystemClock.elapsedRealtime()` for local timers and animations. Use an NTP-derived offset or server timestamps for anything that must be consistent across devices.

</Accordion>
